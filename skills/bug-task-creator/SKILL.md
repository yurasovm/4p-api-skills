# Bug Task Creator

Скилл помогает собрать недостающие данные по ошибке и выдать готовую задачу в понятном формате.

## Когда применять

Применяй этот скилл, если пользователь:
- просит оформить задачу по ошибке;
- дает черновое описание бага;
- просит проверить, это баг или доработка;
- хочет шаблон/финальный текст для баг-трекера.

## Принципы

1. Сначала проверь классификацию: это действительно баг или доработка.
2. Если данных недостаточно, обязательно задавай уточняющие вопросы.
3. Не придумывай факты за пользователя.
4. На выходе всегда давай готовую, структурированную задачу.
5. Пиши кратко и по сути, без воды.

## Критерии: баг vs доработка

Считай **багом**, если:
- было ТЗ/согласованное поведение;
- текущее поведение ему противоречит;
- нужно исправление, а не новое поведение.

Считай **доработкой**, если:
- требования изменились или появились новые;
- пользователь просит новое/измененное поведение;
- нет явного нарушения старых договоренностей.

Если неясно, задай прямой вопрос:
- "Какая договоренность или требование нарушено?"

## Проверка перед созданием задачи

Перед оформлением уточни:
1. Ошибка воспроизводится повторно?
2. Есть ли URL страницы (текстом)?
3. Есть ли шаги воспроизведения?
4. Есть ли скриншоты/видео, где видны весь экран и адресная строка?

Если баг единичный и не повторяется, предложи сначала оформить это как инцидент в техчате, а не как задачу.

## Какие данные обязательно собрать

Собери все обязательные поля:
- Где произошла ошибка (экран/раздел + URL текстом).
- Шаги воспроизведения.
- Фактический результат.
- Ожидаемый результат.
- Почему это ошибка (контекст/бизнес-логика).

Желательные поля:
- Частота (всегда/иногда, N из M попыток).
- Окружение (prod/stage, браузер, устройство, роль пользователя).
- Время и таймзона.
- Вложения (скриншоты, видео, логи, request/response, id сущностей).

## Название задачи

Требования к названию:
- короткое и емкое;
- отражает суть поломки;
- без стоп-слов: "ошибка", "не работает", "проблема".

Примеры хороших названий:
- "Некорректно считается стоимость доставки в корзине"
- "Сортировка товаров не изменяет порядок списка"
- "Не выгружается отчет по товарам"

## Процесс работы

1. Определи: баг или доработка.
2. Если доработка:
   - сообщи, что это доработка, и предложи оформить как feature request.
   - по запросу пользователя помоги сформировать задачу на доработку.
3. Если баг:
   - проверь полноту обязательных полей.
   - задай вопросы только по недостающим данным.
   - после ответов собери финальный bug report.

## Набор уточняющих вопросов (задавай только нужные)

- "На какой странице/экране это произошло? Пришлите точный URL текстом."
- "Какие шаги приводят к ошибке? Опишите по пунктам."
- "Что вы ожидали увидеть?"
- "Что получилось фактически?"
- "Почему это считается ошибкой относительно ожидаемого поведения?"
- "Ошибка воспроизводится стабильно? Как часто?"
- "В каком окружении проверяли (prod/stage), браузер, устройство?"
- "Есть скриншоты/видео? Если да, приложите с видимым URL и отмеченными действиями."

## Формат финального результата

Всегда выдавай результат в этой структуре:

```markdown
### Название
[Короткий и предметный заголовок без стоп-слов]

### Тип
Bug

### Где обнаружено
[Экран/раздел]
URL: [полный URL текстом]

### Предусловия
[Если есть; иначе "Не указаны"]

### Шаги воспроизведения
1. ...
2. ...
3. ...

### Фактический результат
...

### Ожидаемый результат
...

### Обоснование, почему это ошибка
...

### Вложения
- [Скриншот 1]
- [Видео]
- [Логи/ID]
```

## Важные проверки качества

Перед финальной выдачей проверь:
- URL указан текстом и ведет на нужный экран;
- шаги позволяют повторить проблему;
- ожидаемый и фактический результат не перепутаны;
- есть объяснение "почему это ошибка";
- заголовок конкретный и без стоп-слов;
- в тексте нет двусмысленностей вроде "тут ошибка", "неправильно".

Если данных все еще недостаточно, не финализируй задачу: сначала задай недостающие вопросы.
